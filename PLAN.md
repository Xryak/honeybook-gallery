# Honeybook Gallery — Implementation Plan

> A local-only, OTP-gated client gallery view. Backend in Python/FastAPI,
> frontend in Vue 3, glued by a Vite proxy. Built across three sessions:
> two in parallel, one to integrate.

This document is the cold-start brief for each implementation session. If
you are picking this up to implement Session A, B, or C: read the
**Shared context** section, then your session's section. You should not
need to read the other sessions' sections.

The authoritative API contract is [`openapi.yaml`](./openapi.yaml).
Anything in this document that contradicts `openapi.yaml` is a mistake in
this document — fix it.

---

## Shared context (read first, all sessions)

### What we're building

`task.md` in this directory. The flow:

1. Client receives a link `http://localhost:5173/galleries/g_001` (in real
   life, sent via email; here you just type it in).
2. Frontend prompts for a 6-digit OTP. Frontend does NOT auto-trigger OTP
   generation — that's an upstream system in real life. For our mock, the
   user manually runs the CLI (see "OTP CLI" below) to "send" the code.
3. CLI logs the code to backend stdout. User reads it from the terminal
   and types it into the frontend.
4. Frontend posts code to `/api/galleries/g_001/verify`, gets a bearer
   token, stores it (localStorage is fine), navigates to the gallery view.
5. Gallery view fetches `/api/galleries/g_001` with the bearer token,
   renders 10 thumbnails in a grid.
6. Clicking a photo toggles its favorite state via
   `POST /api/galleries/g_001/favourite`. Selected photos are visually
   distinct (e.g., heart icon filled, border highlight, etc. — frontend's
   call).
7. After 10 min, any request returns 401. Frontend clears the token and
   sends user back to OTP entry.

### Sessions

| Session | Owns | Runs |
|---|---|---|
| **A — Backend** | `backend/` directory + `openapi.yaml` (read-only) | In parallel with B |
| **B — Frontend** | `frontend/` directory + `openapi.yaml` (read-only) | In parallel with A |
| **C — Integration** | `vite.config.ts`, root `Makefile`, root `README.md`, frontend tests, end-to-end smoke | After A and B complete |

Session A and Session B never touch each other's directories. They
communicate exclusively through `openapi.yaml`. Session C is the only
session that touches both subtrees.

### Seed data (exact)

Both galleries are created on first backend startup if the DB doesn't
already have them. Re-running the backend is idempotent — it does NOT wipe
favorites.

| Gallery ID | Title | Photos |
|---|---|---|
| `g_001` | "Anna's Wedding" | `p_001` … `p_010` |
| `g_002` | "Marco's Portrait Session" | `p_011` … `p_020` |

Photo files: 20 JPEGs in `backend/seed/photos/p_001.jpg` …
`p_020.jpg`. Generated by Session A using Pillow at first run if absent
— 800×600, a unique hue per photo, the photo ID stamped in the center
in large text. No files checked in.

### Out of scope (explicitly)

Real email, real image upload, rate limiting, OTP attempt counting,
refresh tokens, multi-user per gallery, prod build/deploy, Docker, CI,
HTTPS, CORS (Vite proxy means same-origin).

## Session A — Backend

You own `backend/`. You read `openapi.yaml`. You do not touch `frontend/`.

### Stack

- Python 3.11+
- FastAPI
- SQLAlchemy 2.0 (sync — async would be overkill here)
- pydantic v2 (comes with FastAPI)
- SQLite via `sqlite3` stdlib through SQLAlchemy
- Pillow for seed photo generation
- httpx for the CLI
- Typer or Click for the CLI (your call)
- pytest + `httpx.AsyncClient` or FastAPI's `TestClient` for tests

Linting/formatting/typecheck (ruff, mypy, etc.) is your call.

### Deliverables

1. **`pyproject.toml`** with deps listed above.
2. **FastAPI app** mounted at `/api` for the JSON endpoints and `/static`
   for `seed/`. All four endpoints implemented to match `openapi.yaml`
   exactly — every request shape, every response shape, every status
   code, every error code.
3. **SQLAlchemy models** (suggested — final shape is your call):
   - `galleries(id PK, title)`
   - `photos(id PK, gallery_id FK, url)` — `url` is the relative path
     (e.g. `/static/photos/p_001.jpg`), stored that way in the DB. Never
     store filesystem paths.
   - `favorites(session_id FK, photo_id FK, PK(session_id, photo_id))` —
     favorites are scoped to the session that created them. (See note
     below on idempotency.)
   - `otp_codes(id PK, gallery_id FK, code, expires_at, used_at)`
   - `sessions(token PK, gallery_id FK, expires_at)`
4. **Seeding** runs on app startup. If `galleries` is empty, insert the
   two galleries and their photos. If `backend/seed/photos/` doesn't
   contain the expected JPEGs, generate them with Pillow. Both steps are
   idempotent — re-running adds nothing if already present.
5. **Bearer auth dependency**: a FastAPI dependency that reads
   `Authorization: Bearer <token>`, looks up the session, returns 401
   `{ "error": "expired_session" }` if missing/expired/wrong-gallery, and
   yields the session object otherwise. The dependency must verify that
   the session's `gallery_id` matches the path's `id` — a token for
   `g_001` cannot read `g_002`.
6. **OTP CLI** at `python -m backend.cli send-otp <gallery_id>`:
   - Makes an HTTP POST to
     `http://localhost:8000/api/galleries/<id>/otp` via httpx.
   - Requires the server to be running. If it isn't, print a clear error
     and exit 1.
   - Prints the response and exits 0. (The code itself lands in the
     server's stdout — the CLI does not see it, deliberately.)
7. **pytest suite** covering, at minimum:
   - Each endpoint's happy path.
   - OTP expiry: code stored with `expires_at` in the past → verify
     returns 401.
   - OTP single-use: verify twice with the same code → second one 401.
   - OTP re-request: requesting a new OTP invalidates the prior unused
     one for the same gallery.
   - Auth required: GET `/api/galleries/g_001` without a token → 401.
   - Auth scoped: token for `g_001` used against `g_002` → 401.
   - Favorite toggle: two POSTs to favorite the same photo → final
     `is_favorite` is `false`.
   - Favorite cross-gallery: photo from `g_002` favorited via `g_001`'s
     session → 404 (`not_found`), not 200.

### Notes / non-obvious bits

- **Favorites scope.** The simplest model is: favorites are stored per
  session, so when the session expires the favorites visually disappear
  on the next OTP. That's a defensible mock-grade choice. An alternative
  is per-gallery (anyone authed to `g_001` sees the same favorites) —
  closer to a real shared client view. **Default: per-session.** Document
  whichever you pick in a docstring on the `favorites` model.
- **Time.** Use `datetime.now(timezone.utc)` everywhere for `expires_at`
  comparisons. Store as UTC in SQLite (it'll round-trip as ISO strings,
  that's fine).
- **OTP randomness.** Use `secrets.randbelow(10**6)` and zero-pad — not
  `random.randint`. The mock should still model this correctly.
- **Logging.** The OTP log line must be parseable so the user can read
  the code from a terminal:
  `[OTP] Gallery g_001: 482910 (expires in 10 min)`. Plain `print()` to
  stdout is fine; FastAPI's logger also works.
- **Static mount.** `app.mount("/static", StaticFiles(directory="seed"))`
  served from the backend working directory means run uvicorn from
  `backend/`. Document this in the eventual Makefile target (Session C
  writes the target).

### How you'll run during development

You won't have a Makefile yet — Session C writes it. Your local loop:

```bash
cd backend
pip install -e .
uvicorn app.main:app --reload --port 8000
# In another terminal:
python -m backend.cli send-otp g_001
# Then:
pytest
```

### Definition of done for Session A

- `uvicorn app.main:app` boots without errors on a fresh checkout.
- `python -m backend.cli send-otp g_001` prints a code into the server

- All pytest tests pass.
- `curl` against each endpoint produces responses byte-compatible with

- Re-running the server preserves favorites across restarts.

## Session B — Frontend

You own `frontend/`. You read `openapi.yaml`. You do not touch
`backend/`.

### Stack

- Vue 3 (composition API)
- TypeScript
- Vite
- Vue Router 4
- State: your call (Pinia, composables, plain refs — anything reasonable)
- Styling: your call (plain CSS, CSS modules, Tailwind — anything; the
  spec says "visually distinct selected state," nothing else)
- API types: generate from `openapi.yaml` via `openapi-typescript` (run
  once at setup, commit the output to `src/types/api.ts`). Hand-writing
  is allowed but generation is strictly less error-prone.

Linting/formatting (eslint, prettier) is your call.

### Deliverables

1. **`package.json`** with deps listed above. `npm run dev`, `npm run
   build`, `npm run preview` work.
2. **Routes** (Vue Router):
   - `/galleries/:id` — the only client-facing route. Renders the OTP
     entry component if no valid token is in localStorage, the gallery
     grid otherwise. (You may split this into `/galleries/:id/auth` +
     `/galleries/:id` if you prefer two routes — either is fine.)
   - `/` — landing page. Suggested content: "Open
     `/galleries/g_001` or `/galleries/g_002` to view a gallery." This
     is purely for developer ergonomics.
3. **OTP entry component**:
   - 6-digit numeric input. Either a single masked input or six boxes —
     your call.
   - "Verify" button → `POST /api/galleries/:id/verify { code }`.
   - On 200: store `{ token, expires_at }` in localStorage keyed by
     gallery id (so two galleries' tokens can coexist), transition to
     the gallery view.
   - On 401: show "That code didn't work. Try again." (don't enumerate
     reasons — the API doesn't distinguish).
   - Helpful UX touch: a note like "Your code was logged to the backend
     terminal — copy it from there" so a dev knows where to look. This
     is fine for a mock; remove for production.
4. **Gallery grid component**:
   - On mount: `GET /api/galleries/:id` with the bearer header.
   - Render `photos` as a CSS grid of `<img src="<thumbnail_url>">`
     (the proxy makes these resolve through Vite to the backend).
   - Each photo is clickable. Click → `POST
     /api/galleries/:id/favourite { photo_id }`. On 200, update the
     local photo's `is_favorite` from the response.
   - Selected photos are visually distinct — e.g. heart icon overlay,
     colored border, scale transform. Your call on the exact treatment.
   - On 401 from any call: clear that gallery's token from localStorage
     and route back to OTP entry.
5. **Typed fetch wrapper** in `src/api/`:
   - Always attaches `Authorization: Bearer <token>` if a token exists
     for the current gallery.
   - Maps non-2xx responses to a typed error union (`invalid_code`,
     `expired_session`, `not_found`) using the OpenAPI-generated types.
6. **No tests.** Session C adds Vitest + (optional) Playwright.

### How you'll run during development

Vite dev server defaults to port 5173. You don't have the proxy yet —
Session C writes it. For your dev loop, pick one:

- **Option 1: stand up Session A's backend** if it's ready first. You
  can hit `http://localhost:8000/api/...` directly during dev by
  temporarily putting an absolute base URL in your fetch wrapper, then
  removing it before handoff. (Cleanest.)
- **Option 2: mock with [msw](https://mswjs.io)**. Spin up MSW handlers
  matching `openapi.yaml`. Lets you develop fully offline. Slight
  duplication of the contract.

Either way: your committed fetch wrapper uses **relative URLs only**
(`/api/galleries/...`). The dev-time base URL trick stays in a `.env` or
a constant you remove.

### Definition of done for Session B

- `npm run dev` boots without errors on a fresh `npm install`.
- The two screens render and look reasonable.
- The TS types in `src/types/api.ts` are generated from `openapi.yaml`
  and used in the fetch wrapper.
- localStorage token persistence works across page reloads.
- 401 from any call routes back to OTP entry and clears the token.
- All fetches use relative URLs (`/api/...`, `/static/...`).

### Likely fix list during integration

These are the contract edges most likely to drift between Session A
and Session B. Verify each before declaring done:

- Photo URL paths exactly match (`/static/photos/p_001.jpg` everywhere,
  not `/photos/p_001.jpg` or `/static/p_001.jpg`).
- Bearer header capitalization (`Authorization`, `Bearer`).
- Error envelope shape (`{ error: "..." }`, not `{ message: "..." }` or
  `{ detail: "..." }`).
- Status code on auth failure (401, not 403).
- 200 with `is_favorite` field on the toggle response (not 204 or 200
  with empty body).
- Body of `/otp` is empty, not `{}` — both should work but flag if the
  backend requires `{}`.

If you fix a drift on one side, also fix it on the other if needed, then
update `openapi.yaml` if the contract itself was ambiguous.

## Glossary

- **OTP** — one-time password, the 6-digit code sent (logged) on the
  `/otp` endpoint.
- **Bearer token** — opaque random string returned by `/verify`,
  presented as `Authorization: Bearer <token>` on subsequent requests.
- **Session** — a row in the `sessions` table tying a token to a
  `gallery_id` with an `expires_at`.
- **Gallery scope** — both OTPs and sessions are scoped to a specific
  gallery. Authenticating to one gallery does not give access to
  another.
- **"Send" an email** — log the OTP code to backend stdout. No SMTP, no
  network calls.
